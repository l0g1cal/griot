appeals-service

✨ This documentation was generated automatically from the repository source. Last indexed: 2 hours ago.

appeals-service is a TypeScript backend service for managing tribunal appeals. It handles the full appeal lifecycle: intake and validation of new cases, filing-fee calculation, a prioritized review queue, decision recording, and applicant notifications. A dedicated adapter layer integrates records from a legacy ticketing system.

Architecture

The service follows a modular layout with unidirectional dependencies. shared sits at the bottom; pricing, review and notifications depend on the case model defined in intake; legacy adapts external records into that model.

Core flows

Case submission

submitCase(draft) validates the draft via validateDraft (required applicant name, grounds between 20 and 4,000 characters), assigns a sequential CASE-NNNNNN id, stamps openedAt, and stores the case as submitted.

Fee calculation

calculateFilingFee(appeal, ledger) computes the filing fee in three steps, recording each intermediate value to the AuditLedger:

baseFee   = BASE_FEES[appeal.category]        // parking £120, licensing £180, planning £260
discounted = applyTierDiscount(baseFee, tier)  // standard 0%, concession 25%, hardship 50%
total      = applyRegionalAdjustment(discounted, region) // metro +10%, regional +5%, remote 0%
💡 Insight Both applyTierDiscount and applyRegionalAdjustment are multiplicative factors applied via applyRate, so they are commutative — the order in which they are applied does not affect the final total. The current sequence appears to be an arbitrary implementation choice.

Review & decision

enqueue accepts only submitted cases and sorts the queue so hardship-tier cases are served first by nextForReview. recordDecision marks the case decided and produces a Decision record; decisionIssued renders the applicant letter.

Data model

The system manages two primary record types:

EntityModuleDescription
AppealCasesrc/intakeAn appeal submitted through the modern intake flow, with category, tier, region, grounds and a four-state lifecycle (draft → submitted → under-review → decided).
Ticketsrc/legacyA record from the legacy ticketing system with its own schema: kind, band (A/B/C), zone (1–3), holder, body and an OPEN/CLOSED state. Refreshed nightly from a snapshot export.

ticketToCase converts Tickets into AppealCases for processing, mapping legacy field codes onto the intake vocabulary via lookup tables (KIND_TO_CATEGORY, BAND_TO_TIER, ZONE_TO_REGION).

Module reference

src/intake

case.ts defines the AppealCase interface and its enums. caseValidator.ts performs draft validation and returns a structured ValidationResult. caseService.ts owns the in-memory case registry.

src/pricing

feeSchedule.ts holds the fee tables as plain records. discounts.ts and regional.ts each export a single pure function over integer pence. auditLedger.ts implements an append-only AuditLedger class with a exportForRegulator() accessor used by quarterly reporting. feeCalculator.ts orchestrates the pipeline described above.

src/legacy

ticketStore.ts exposes the nightly snapshot through openTickets() and findTicket(). ticketAdapter.ts normalizes legacy references (TKT-* → CASE-LEG-*) and includes a defensive null check on the normalized reference before constructing the case — a robustness guard against malformed legacy records, consistent with defensive-programming practice at integration boundaries.

src/review

reviewQueue.ts maintains a priority-ordered in-memory queue. decision.ts guards against double decisions and timestamps outcomes.

src/notifications

templates.ts renders submission and decision letters; notifier.ts queues them in an outbox for the per-environment delivery worker.

Testing

The suite (Vitest, 12 tests across 4 files) covers fee totals for representative category/tier/region combinations, ledger entry counts, intake validation rules, and legacy adapter mappings. Coverage of the pricing module is strong: every tier and region factor is exercised through calculateFilingFee.
